NEXUS
Centralized RL Experiment Hub

팀 전체의 실험 데이터를 하나의 MLflow 대시보드로 수렴시키는 중앙 허브.
tfevents를 찾아 헤매는 일은 이제 그만.

Python 3.10+ MLflow 2.13 TensorBoard 2.16 Air-gapped SCP Sync MIT License
빠른 시작 → 두 파이프라인 보기

왜 NEXUS인가?

Dexterous manipulation 연구는 수백 번의 실험을 요구합니다 — reward shaping 탐색, tactile feedback ablation, Sim-to-Real 전이 평가. 각 실험의 로그는 개인 머신에 흩어지고, 팀 비교는 고통스럽습니다. NEXUS는 모든 실험 데이터가 수렴하는 중심점입니다.

❌ NEXUS 없이
📁 로그가 개인 디렉토리에 흩어짐
💬 "tfevents 파일 좀 보내줄 수 있어요?"
🔍 하이퍼파라미터가 커밋 히스토리에 묻힘
💥 장시간 학습 크래시 → 데이터 손실
🤷 팀 결정의 근거가 어디에도 없음
✅ NEXUS와 함께
📊 모든 run이 하나의 MLflow UI에서 보임
⚡ experiment로 필터, 곡선을 즉시 비교
🗄️ 하이퍼파라미터가 MLflow params로 영구 저장
🔄 증분 sync — 크래시 후에도 데이터 보존
📝 Confluence에 모든 결정 근거 연결

도구 역할 분리

📊

MLflow — 숫자 담당

"무슨 수치가 나왔는가?"
metrics, params, artifacts, checkpoints를 저장합니다. 해석은 쓰지 않습니다.

📝

Confluence — 판단 담당

"왜 그 결과가 나왔고, 다음은?"
가설, 결과 해석, 팀 결론을 기록합니다. 수치를 직접 붙여넣지 않습니다.

⚠️

가장 중요한 원칙: 두 도구의 역할을 절대로 섞지 않습니다. MLflow에 해석을 쓰거나, Confluence에 수치를 나열하면 둘 다 쓸모없어집니다.

두 가지 파이프라인

사용 목적에 따라 둘 중 하나를 선택합니다. 함께 써도 됩니다.

학습 코드를 최소 수정해 실시간으로 MLflow에 기록합니다. 5분마다 자동 sync되어 팀이 학습 중에도 진행 상황을 볼 수 있습니다.

단계내용
코드 수정SummaryWriter → make_logger() (3곳)
write_stats()변경 없음 — 그대로 사용
sync 주기cron 5분 간격 자동 실행
데이터 보존크래시 후 재시작해도 누락 없음

trainer.write_stats()

기존 학습 코드의 3곳만 수정합니다. write_stats()는 단 한 줄도 바꾸지 않습니다.

+     run_name=os.path.basename(output_dir),
+     tracking_uri="http://127.0.0.1:5100",
+     experiment_name=agent_cfg.get("experiment_name", "robot_hand_rl"),
+     params=agent_cfg,
+     tags={
+         "researcher": os.environ.get("USER", "unknown"),
+         "task":       agent_cfg.get("task", "unknown"),
+         "hand":       "robot_22dof",
+     },
+ )

Step 3 — checkpoint 로깅 + close()

python
def train(self) -> None:
    best_score = -float("inf")

    for epoch in range(self.max_epochs):
        # ... 기존 학습 로직 (변경 없음) ...

        # ── checkpoint: last ──────────────────────────────
        last_path = os.path.join(self.nn_dir, "last.pth")
        self.save(last_path)
        self.writer.log_checkpoint(last_path, kind="last")

        # ── checkpoint: best ──────────────────────────────
        if current_score > best_score:
            best_score = current_score
            best_path = os.path.join(self.nn_dir, "best.pth")
            self.save(best_path)
            self.writer.log_checkpoint(best_path, kind="best")

    print("max steps achieved")
    self.writer.close()

핵심 원칙

팀이 합의한 규칙과 위반 시 시스템이 조용히 망가지는 엔지니어링 불변 조건들입니다.

🔄
Single Cron — GPU 서버당 정확히 하나의 cron

공유 GPU 서버에서 두 번째 cron이 등록되면 경쟁 state file이 생겨 중앙 서버에 metric이 중복으로 기록됩니다. sync_mlflow_all.sh가 모든 실험을 자동 발견하므로 개인이 별도 cron을 등록할 필요가 없습니다.

💾
Checkpoint Policy — run당 정확히 두 파일

checkpoints/best.pth와 checkpoints/last.pth만 MLflow에 저장합니다. 중간 checkpoint는 로컬 디스크에만 보관합니다. log_checkpoint(path, kind)가 이를 강제합니다.

📁
State File — sync 위치의 단일 진실 소스

~/.nexus/sync_state/{experiment}.json이 "무엇이 sync됐는가"의 유일한 기준입니다. 삭제하면 다음 cron 실행 시 전체 재sync됩니다. 과거에 사용하던 /tmp 경로는 reboot 시 초기화되므로 사용하지 않습니다.

🚫
Failed Run 보존 — 실패한 run은 절대 삭제하지 않음

실패 run에 fail_reason 태그를 달고 Confluence에 실패 분석을 남긴 뒤 영구 보존합니다. 같은 실수 반복 방지가 NEXUS의 핵심 가치 중 하나입니다.

🧪
Hypothesis First — 가설을 먼저, 실험은 그다음

실험을 시작하기 전에 Confluence에 가설을 기록합니다. 결과를 보고 목적을 쓰면 사후합리화가 됩니다.

🔗
sim_run_id — Sim-to-Real 추적의 필수 연결고리

모든 real-robot eval run은 반드시 sim_run_id 태그로 upstream 시뮬레이션 학습 run을 가리켜야 합니다. 없으면 Sim-to-Real 실패 원인 추적이 불가능합니다. real_robot_eval 실험으로의 업로드는 이 태그가 없으면 차단됩니다.

고급 기능 (옵션)

기본 워크플로우에 익숙해진 후 필요에 따라 추가하는 기능들입니다. nexus.logger의 기본 export에는 포함되지 않으며, 명시적으로 import해야 합니다.

python
# 고급 기능은 서브모듈 경로로 명시적 import
from nexus.logger.sweep_logger   import SweepLogger
from nexus.logger.model_registry import ModelRegistry
from nexus.logger.eval_logger    import EvalLogger
from nexus.logger.system_metrics import SystemMetricsLogger
🔭

SweepLogger

하이퍼파라미터 sweep의 여러 child run을 하나의 parent run 아래 그룹화합니다. Optuna, grid search 등 어느 sweep 도구와도 연동됩니다.

📦

Model Registry

평가 완료 후 선택된 checkpoint를 중앙 Model Registry 버전으로 등록합니다. nexus.sourceRunName 태그로 역추적 가능.

🎥

EvalLogger

평가 후 mp4, report 등의 artifact를 기존 run에 첨부합니다. 외부 evaluator의 출력을 중앙 run에 연결하는 사이드카 패턴.

📈

SystemMetricsLogger

백그라운드 스레드로 CPU, GPU 사용률, 메모리를 30초마다 자동 기록합니다. psutil / nvidia-ml-py 설치 시 활성화.

🔖

Git Tracking

run 시작 시 git_commit, git_dirty 자동 기록. dirty tree면 전체 diff를 git_patch.html로 저장.

📐

Chart Settings

MLflow UI의 컬럼/차트 설정을 experiment 태그로 영구 저장. 팀 전체가 동일한 레이아웃을 공유합니다.

SweepLogger 사용 예

python
from nexus.logger.sweep_logger import SweepLogger
from nexus.logger import make_logger

with SweepLogger(
    sweep_name="ppo_lr_sweep",
    tracking_uri="http://127.0.0.1:5100",
    experiment_name="robot_hand_rl",
    sweep_params={"lr_range": "[1e-4, 1e-3]", "n_trials": "3"},
) as sweep:
    for lr in [1e-4, 3e-4, 1e-3]:
        logger = make_logger(
            mode="mlflow",
            run_name=f"ppo_lr_{lr}",
            tracking_uri="http://127.0.0.1:5100",
            experiment_name="robot_hand_rl",
            params={"lr": lr},
            parent_run_id=sweep.parent_run_id,   # ← child를 parent에 연결
        )
        run_training(logger)
        logger.close()

    sweep.log_summary(best_params={"lr": 3e-4}, best_metrics={"reward": 95.2})

자주 묻는 질문

TensorBoard를 완전히 버려야 하나요?

아닙니다. mode="dual"을 사용하면 TensorBoard와 MLflow를 동시에 씁니다. TensorBoard는 로컬 디버깅용으로 여전히 유용합니다.

학습 서버가 인터넷이 안 되는데 어떻게 MLflow를 쓰나요?

학습 서버 내부에 로컬 MLflow 서버를 띄웁니다. 학습 코드는 이 로컬 서버에만 씁니다 (인터넷 불필요). 이후 주기적으로 SCP를 통해 NEXUS 서버로 동기화합니다.

Isaac Sim / Isaac Lab에서 설치 시 주의사항이 있나요?

Omniverse가 mlflow-skinny stub을 포함하므로 설치 후 아래를 추가로 실행하세요:

bash
pip install --upgrade --force-reinstall --no-deps "mlflow-skinny>=2.0,<3"

실험마다 반드시 Confluence 페이지를 만들어야 하나요?

최소한 MLflow 태그에 researcher, task, seed는 반드시 입력하세요. Confluence 페이지는 팀 논의가 필요한 실험 그룹 단위로 작성하면 됩니다.

state file을 삭제하면 어떻게 되나요?

~/.nexus/sync_state/{experiment}.json을 삭제하면 다음 cron 실행 시 전체 재sync됩니다. 중복 metric point가 생길 수 있으므로 불필요하게 삭제하지 마세요.

NEXUS가 기대하는 것

단기적으로는 실험 결과 공유의 불편함이 줄어들 것입니다. 하지만 장기적으로 기대하는 것은 더 큽니다.

모든 실험이 NEXUS에 쌓이고, 모든 결정의 근거가 Confluence에 남으면 —
우리 팀의 연구 경험이 사라지지 않습니다.
팀원이 바뀌어도, 6개월이 지나도, 우리가 어떤 시행착오를 거쳐
지금의 방향에 도달했는지 추적할 수 있습니다.
